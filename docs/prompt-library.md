# Prompt Library For Product Designer Portfolio

## Roadmap 1–10

| # | Prompt | Goal | Priority | When |
| --- | --- | --- | --- | --- |
| 1 | Site audit | Understand structure, weaknesses, and redesign opportunities. | High | Project start |
| 2 | USP / hero | Lock positioning and value proposition. | High | Right after audit |
| 3 | Case structure | Build a reusable skeleton for project stories. | High | Before long-form writing |
| 4 | Full case article | Turn project facts into a publishable case study. | Very high | After structure and facts |
| 5 | About + skills | Add context and concise self-presentation. | Medium | In parallel with case work |
| 6 | Visuals + wireframes | Visualize covers and page layouts fast. | Low | After content is shaped |
| 7 | Metrics + KPI | Strengthen cases with proof and business effect. | High | Before final case draft |
| 8 | SEO + metadata | Prepare discoverability and accessibility basics. | Medium | Before publishing |
| 9 | QA checklist | Catch issues before release. | High | Final pre-release pass |
| 10 | Post-release validation | Test conversion hypotheses and collect feedback. | Medium | After publishing |

Recommended order: `1 → 2 → 3 → 7 → 5 → 4 → 6 → 8 → 9 → 10`

## Workflow

1. Run prompts `1–3` to audit the current site, define positioning, and create a case skeleton.
2. Continue with `2 → 3 → 7 → 5 → 4 → 6 → 8 → 9 → 10`.
3. Manually refine every output: verify facts, remove generic wording, add real project details.
4. Combine outputs into the site: USP in hero, full case in project page, metrics in results block, metadata in `<head>`, alt texts in images.
5. Use prompt `9` for final QA and prompt `10` for post-release validation through A/B tests and recruiter feedback.

## 1. Site Audit

```text
Ты — UX-специалист и аудитор портфолио-сайтов. Проведи всесторонний аудит сайта {url} с учётом тематики и ключевых слов: {keywords}.

Проверь:
1. Структуру сайта
2. Навигацию
3. Hero-блок
4. Кейсы / работы
5. CTA
6. SEO
7. Скорость и техническое впечатление
8. Адаптивность
9. Доступность
10. Общее позиционирование

Формат ответа:
- Пиши кратко, по делу, в Markdown
- Для каждого раздела дай:
  - 1–3 сильные стороны
  - 1–3 слабые стороны
  - короткие рекомендации
- В конце добавь:
  - Общую оценку сайта по 10-балльной шкале
  - Топ-5 приоритетных улучшений
  - Что критично исправить в первую очередь перед редизайном

Если какие-то данные нельзя надёжно определить без доступа к коду или Lighthouse, так и укажи: “требует дополнительной проверки”.
```

## 2. USP / Hero

```text
Ты — UX-копирайтер и бренд-стратег для портфолио продуктовых дизайнеров. Сгенерируй уникальное торговое предложение для портфолио на основе входных данных.

Входные данные:
- Специализация: {specialty}
- Опыт: {experience}
- Сферы: {domains}
- Уровень: {mid_or_senior}

Требования:
- Сгенерируй 3 варианта УТП:
  1. Короткое: 1 предложение
  2. Среднее: 2–3 предложения
  3. Длинное: 1 абзац
- Если уровень = senior, делай акцент на системном мышлении, влиянии на продукт и бизнес-результат
- Если уровень = mid, делай акцент на сильной практике, качестве исполнения и умении решать продуктовые задачи
- Избегай пустых формулировок и общих клише
- Язык ответа: русский

Формат ответа:
- markdown-список из 3 пунктов
- У каждого пункта должен быть заголовок:
  - Короткое
  - Среднее
  - Длинное
- Не пиши пояснений, только готовые варианты УТП
```

## 3. Case Structure

```text
Ты — UX-редактор и product design mentor. Разбей кейс продуктового дизайна на логические разделы для портфолио.

Входные данные:
- Название проекта: {project_title}
- Роль: {role}

Обязательные разделы:
1. Название проекта и роль
2. Задача / проблема
3. Исследование
4. Решение / прототипы
5. Результат и метрики
6. Выводы

Для каждого раздела укажи:
- название раздела
- что именно в нём должно быть
- в каком формате лучше подать информацию
- что особенно важно показать, чтобы кейс выглядел убедительно

Формат ответа:
- markdown-список из 6 пунктов
- В каждом пункте:
  - Название раздела
  - Короткое пояснение
  - Рекомендуемый формат подачи
```

## 4. Full Case Article

```text
Ты — UX-редактор и senior product design storyteller. На основе входных данных составь полный кейс продуктового дизайна в формате статьи для портфолио.

Входные данные:
- Название проекта: {project_title}
- Роль: {role}
- Размер команды: {team_size}
- Длительность проекта: {duration}
- Проблема: {problem}
- Инструменты: {tools}
- Целевая аудитория: {target_users}
- Решение: {solution}
- Метрики до: {metrics_before}
- Метрики после: {metrics_after}

Структура кейса:
1. Обзор
2. Задача
3. Исследование
4. Решение
5. Результаты
6. Выводы

Требования:
- Пиши на русском языке
- Оформляй ответ в Markdown
- Используй заголовки и при необходимости маркированные списки
- Не выдумывай факты, если данных недостаточно
- В разделе «Результаты» сравни показатели до и после, если они переданы
- В разделе «Решение» делай акцент на продуктовой логике, сценариях и итерациях
- В разделе «Выводы» укажи уроки, ограничения и возможные следующие шаги
```

## 5. About + Skills

```text
Промпт A. Ты — UX-копирайтер для портфолио продуктовых дизайнеров. Создай текст для раздела «О себе».

Входные данные:
- Опыт: {experience_years}
- Роли / компетенции: {roles}
- Что важно в дизайне: {design_values}

Требования:
- Объём: 3–5 предложений
- Пиши от первого лица
- Упомяни опыт, ключевые компетенции и отношение к дизайну
- Сделай акцент не только на визуальной части, но и на решении продуктовых задач
- Формат ответа: один абзац без заголовка

Промпт B. Ты — карьерный редактор и UX-копирайтер. Сгенерируй список ключевых навыков продуктового дизайнера на основе опыта и инструментов.

Входные данные:
- Опыт: {experience_years}
- Инструменты: {tools}

Требования:
- Вывод в формате markdown-списка
- Включай не только инструменты, но и реальные навыки
- Учитывай опыт: чем он выше, тем сильнее акцент на системное мышление и влияние на продукт
- Формат: 8–12 пунктов без вступления и заключения
```

## 6. Visuals + Wireframes

```text
Промпт A. Ты — арт-директор и product design visual editor. Придумай визуальную обложку для кейса продуктового дизайнера.

Входные данные:
- Название проекта: {project_title}
- Контекст проекта: {project_context}
- Брендовые цвета: {brand_colors}

Опиши:
- общую визуальную идею
- композицию
- фон
- главный объект или графический акцент
- цветовую палитру
- типографику
- короткий слоган

Формат ответа:
- markdown
- разделы:
  - Концепция
  - Композиция
  - Цвета
  - Типографика
  - Слоган
  - Примечания

Промпт B. Ты — UX-дизайнер и information architect. Создай wireframe главной страницы портфолио продуктового дизайнера в формате ASCII или Mermaid.

Входные данные:
- Цвета бренда: {brand_colors}
- Шрифт: {font}

На странице обязательно должны быть:
- хедер с именем и навигацией
- hero-блок с коротким позиционированием
- блок с избранными кейсами
- секция «О себе»
- блок навыков или экспертизы
- CTA-блок для связи
- футер

Промпт C. Ты — mobile UX designer. Создай упрощённый прототип мобильной версии портфолио размером 375×812 px в формате ASCII.

Что должно быть:
- компактный хедер
- burger / swipe-меню
- hero-блок
- вертикальный список кейсов
- краткий блок «О себе»
- CTA для связи
- футер
```

## 7. Metrics + KPI

```text
Ты — product analyst и UX-редактор. На основе качественных результатов проекта предложи подходящие количественные метрики, которые можно использовать в кейсе продуктового дизайнера.

Входные данные:
- Название проекта: {project_title}
- Качественные результаты: {qualitative_outcomes}

Требования:
- Подбирай только те метрики, которые действительно связаны с описанными результатами
- Для каждой метрики укажи:
  - название
  - что она измеряет
  - почему она подходит
  - простую формулу или способ расчёта
- Если точные цифры неизвестны, не выдумывай значения

Формат ответа:
- markdown bullet list
- 1–2 пункта
- каждый пункт в формате:
  - Название метрики
  - Описание
  - Почему подходит
  - Формула / способ расчёта
```

## 8. SEO + Metadata

```text
Промпт A. Ты — SEO-редактор и UX-копирайтер для портфолио продуктовых дизайнеров. Сгенерируй метаданные для страницы портфолио.

Входные данные:
- Специализация: {specialty}
- Пример проекта: {project_title}
- Имя дизайнера: {designer_name}
- Язык страницы: {page_language}

Нужно сгенерировать:
- title
- meta description
- keywords

Требования:
- title до 60 символов
- description до 155 символов
- keywords — несколько релевантных фраз через запятую
- Формат ответа:
  <title>...</title>
  <meta name="description" content="...">
  <meta name="keywords" content="...">

Промпт B. Ты — UX-редактор и SEO-специалист. Для изображений кейсов сгенерируй короткие и полезные alt-тексты.

Входные данные:
- Список файлов: {image_list}
- Контекст кейса: {project_context}
- Тип изображений: {image_type}

Требования:
- Длина каждого alt-текста: 5–10 слов
- Формат ответа: JSON
- Ключ — имя файла, значение — alt-текст
```

## 9. QA Checklist

```text
Ты — QA-специалист и UX-аудитор. Составь чек-лист перед публикацией портфолио-сайта продуктового дизайнера.

Входные данные:
- Список страниц: {pages}

Что нужно проверить:
1. Скорость и производительность
2. Адаптивность
3. Доступность
4. Ссылки и навигацию
5. CTA и формы
6. Контент и визуальную консистентность
7. SEO и метаданные
8. Базовую техническую корректность

Требования:
- Формат ответа: markdown-список
- Группируй пункты по разделам проверки
- Пиши короткие, проверяемые и конкретные пункты
- Не добавляй вступление и заключение
```

## 10. Post-Release Validation

```text
Ты — product strategist и UX researcher. Составь план проверки эффективности портфолио после публикации.

Входные данные:
- Цель сайта: {goal}
- Основные CTA: {cta_list}
- Ключевые страницы: {pages}

Сформируй:
1. 3 гипотезы для A/B-тестирования
2. Для каждой гипотезы:
   - что меняем
   - на какой странице
   - какую метрику отслеживаем
   - какой сигнал будет считаться улучшением
3. Короткий список вопросов для 5 интервью с рекрутерами или hiring managers
4. Рекомендации, как интерпретировать результаты без ложных выводов

Формат ответа:
- Markdown
- Блоки: Гипотезы, Метрики, Интервью, Как принимать решения
- Без воды и общих формулировок
```
